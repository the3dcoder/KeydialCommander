# Phase 1.5 — evdev-grab Device Layer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (or subagent-driven-development) to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Replace the non-functional bleak/FFE1 input path with a Linux **evdev-grab + uinput** device layer that actually receives the K20's button and dial input (live-verified in `docs/DEVICE-K20.md` §G), keeping every downstream Phase 1 component (ProfileStore, KeybindManager, UInputHandler, EventBus, IPC, CLI) unchanged.

**Architecture:** The daemon opens the K20's kernel evdev nodes (`event26` keyboard, `event27` mouse), `EVIOCGRAB`s them (suppressing the original keystrokes), reads decoded `KEY_*`/`REL_*` events, maps them to action IDs (`BUTTON_1..18`, `DIAL_CW/CCW/CLICK`), runs the existing combo/sticky logic, looks up bindings, and emits via the existing `UInputHandler`. USB and Bluetooth use the same nodes. bleak, `hid_parser`, `bluetooth_watcher`, and the `hid-generic` unbind subsystem are retired.

**Tech Stack:** Python ≥3.8, `python-evdev` (already a dependency), asyncio, `pyudev` (new, for hotplug — with a periodic-rescan fallback if unavailable), pytest.

## Global Constraints

- Python floor 3.8: `typing.Optional/List/Dict`, no `X | Y`, no `match`.
- Never block the event loop; evdev reads use `async_read_loop()`.
- The device is identified by evdev **name matching `*Keydial*`** and/or **VID:PID `0x256C:0x8251`** (bustype BT or USB) — never by fixed `eventNN` numbers (they change per connect).
- Downstream interfaces are **frozen**: `UInputHandler.send_event(InputEvent)`, `KeybindManager.get_action(action_id)`, `EventBus.publish(dict)`, `ProfileStore` — do not change their signatures.
- `InputEvent`/`EventType` move to a new `input_events.py`; `uinput_handler.py` imports from there. `event_type` is `KEY_PRESS`/`KEY_RELEASE`; `key_code` is the action ID.
- Access requires the `70-huion-keydial-mini.rules` uaccess rule (event nodes + `/dev/uinput`) — already on `main`.
- All work on branch `feat/evdev-device-layer`; `.venv/bin/pytest tests/ -q` must stay green.
- Commit messages end with `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.

## Live-verified input map (DEVICE-K20 §G) — the source of truth for `input_map.py`

`KEY_K→BUTTON_1, KEY_G→BUTTON_2, KEY_L→BUTTON_3, KEY_DELETE→BUTTON_4, KEY_I→BUTTON_5, KEY_D→BUTTON_6, KEY_B→BUTTON_7, KEY_E→BUTTON_8, KEY_S→BUTTON_9, KEY_Z→BUTTON_10, KEY_C→BUTTON_11, KEY_V→BUTTON_12, KEY_LEFTCTRL→BUTTON_13, KEY_LEFTALT→BUTTON_14, KEY_LEFTSHIFT→BUTTON_15, KEY_ENTER→BUTTON_16, KEY_SPACE→BUTTON_17, KEY_N→BUTTON_18, KEY_PLAYPAUSE→DIAL_CLICK`; `REL_WHEEL -1 → DIAL_CW`, `REL_WHEEL +1 → DIAL_CCW` (sign confirmed at implementation time against the live device; a single `INVERT_WHEEL` constant makes it trivial to flip).

### Task 4: `evdev_source.py` — grab, read, hotplug

**Files:** Create `src/huion_keydial_mini/evdev_source.py`; Test `tests/test_evdev_source.py`

**Interfaces produced:**
- `class EvdevSource:`
  - `__init__(self, on_event: Callable[[InputEvent], Awaitable] , on_state: Callable[[bool], None] = None)`
  - `async start(self)` — discover Keydial nodes (`evdev.list_devices()` filtered by `is_keydial`), open + `EVIOCGRAB` each, launch an async read task per node; start hotplug watch
  - `async stop(self)` — ungrab, close, cancel tasks
  - internal: each read task converts evdev events via `input_map` and calls the translator, then `on_event` for each produced `InputEvent`; wheel hi-res (`REL_WHEEL_HI_RES`) is ignored (use `REL_WHEEL` ±1 detents); `KEY_*` repeat events (value 2) are ignored
  - hotplug: a `pyudev` monitor on the `input` subsystem re-runs discovery on add/remove; if `pyudev` import fails, fall back to a 2 s periodic rescan
- Grab safety: on `start`, if grab raises (device busy), log and retry with backoff; never crash the daemon.
