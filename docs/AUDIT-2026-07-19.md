## ⚠ ADDENDUM (2026-07-19 evening) — CRITICAL: wrong input channel

**Finding C1 (critical, live-verified):** The entire driver reads input from the wrong place.
The K20 "Keydial mini-504" (firmware `HUION_T21h_230628`) is a **standard BLE HID keyboard**.
Its button/dial input flows through the standard HID service `0x1812` → BlueZ HoG → kernel evdev,
a path that is **never exposed on D-Bus GATT**. The vendor `FFE1` characteristic the driver
subscribes to is a firmware/status channel and emits **zero input** (its own descriptor literally
reads `HUION_T21h_230628`; the sibling vendor char is labelled `OTA`). Verified three ways:
(a) direct D-Bus subscription to all notify characteristics — only battery/status frames, no
input; (b) the real daemon connected, subscribed to FFE1, received nothing on keypress;
(c) reading the kernel evdev nodes — every button and the dial appear there as clean `KEY_*`/
`REL_*` events. Full map in `docs/DEVICE-K20.md` §E–G.

**Consequence:** the original `bleak`→`FFE1`→`hid_parser` device layer cannot receive input on
this hardware — it never worked. The fix is an architecture pivot to **evdev-grab + uinput**
(the `keyd`/`xremap`/`input-remapper` pattern), planned as Phase 1.5
(`docs/superpowers/plans/2026-07-19-phase1.5-evdev-device-layer.md`). Everything else the audit
covers below — and all of Phase 1's control plane (profiles, persistence, IPC, events,
validation, uinput emit, CLI, packaging) — remains valid; only `device.py` + `hid_parser.py` are
replaced. This supersedes DEVICE-K20 §E/F's original FFE1 conclusion and audit findings that
assumed the BLE path (H2 device-identity, M9 hci-adapter, the udev unbind rule M4) — those become
moot in the evdev architecture.

## 1. What this project is

A ~2,800-line Python (3.8+) user-space **Bluetooth-only** driver for the Huion Keydial Mini (K20), MIT-licensed. It bypasses the kernel's default treatment of the device (a plain HID keyboard with fixed firmware shortcuts) and re-maps everything in user space:

```
BlueZ D-Bus signal (device connected)
  → bleak GATT client subscribes to all notify characteristics
    → hid_parser.py decodes fixed-format reports
        (type-1 keycodes bytes 3–5, type-2 modifier bitmask byte 0, dial reports prefixed 0xf1)
      → action IDs: BUTTON_1..18, chords "BUTTON_1+BUTTON_2" (peak-button-set), DIAL_CW/CCW/CLICK
        → keybind_manager.py resolves the action (in-memory map; Unix-socket JSON control)
          → uinput_handler.py emits mapped EV_KEY events via a virtual evdev device
```

| Component | File | Responsibility |
|---|---|---|
| Entry/lifecycle | `src/huion_keydial_mini/main.py` | CLI (click), DriverManager, signals, 1 s poll loop |
| BLE device | `src/huion_keydial_mini/device.py` | connect/retry, notification fan-in, teardown |
| Connection watcher | `src/huion_keydial_mini/bluetooth_watcher.py` | raw D-Bus AddMatch on BlueZ `PropertiesChanged` (`Connected`) |
| Report decoding | `src/huion_keydial_mini/hid_parser.py` | report → InputEvent; combo + sticky state machines |
| Binding store | `src/huion_keydial_mini/keybind_manager.py` | in-memory map + asyncio Unix-socket server + client helper |
| Virtual device | `src/huion_keydial_mini/uinput_handler.py` | 225-entry KEY_MAPPING, capability build, EV_KEY emission |
| CLI client | `src/huion_keydial_mini/keydialctl.py` | bind/unbind/list/reset/set-device/clear-device |
| Config | `src/huion_keydial_mini/config.py` | YAML load/validate, flat→nested migration, save |
| Debug logger | `src/huion_keydial_mini/event_logger.py` | human-readable event stream, `--test` mode |

Actions today are **keyboard-type only** (`EventType.KEYBOARD`); binding `BTN_LEFT`-style mouse *clicks* works end-to-end, but mouse movement/scroll and "mixed combo" action types exist only in the README.

**Fixed firmware mapping discovered:** the K20's stock firmware sends ordinary keyboard usages — the parser's scancode table corresponds to `K G L Del I D B E S Z C V` (buttons 1–12), modifier bits `Ctrl/Alt/Shift` (buttons 13–15), and `Enter Space N` (buttons 16–18). Buttons 13–15 arrive in the modifier bitmask (byte 0), which is why they're "type 2".

## 2. Live hardware state (this machine, 2026-07-19)

- Ubuntu 24.04.4, kernel 6.17, GNOME on **Wayland**.
- K20 connected via **Bluetooth** as `Keydial mini-504` (MAC `20:23:06:01:8A:B0`), battery 90%, Appearance 0x03C1 (keyboard), Modalias `usb:v256Cp8251d0001`.
- GATT services advertised: 1800/1801, 180A Device Info, 180F Battery, **1812 HID**, vendor `0000ffe0-…` and `00010203-0405-0607-0809-0a0b0c0d1912` (Huion/UGEE vendor service — carries LED/sleep/init commands in Huion's ecosystem).
- Kernel side: two uhid devices (`0005:256C:8251.0011/.0012`) → input nodes "Keydial mini-504 Keyboard" and "…Mouse", both bound to `hid-generic`. **The driver in this repo is not installed/running here** — the device currently types its fixed firmware shortcuts.
- **USB is charge-only right now**: no `256c` entry in `lsusb`. Per the official manual, USB-C *is* a data connection with the right cable — the present cable/port combination doesn't enumerate.
- Official-manual facts relevant to the roadmap: LED brightness (5 levels) and sleep timer (15 m/30 m/1 h/1.5 h) are driver-settable (vendor commands, unimplemented here); Huion's own driver implements *press-dial-to-cycle-3-dial-functions* in software; force-pairing = hold a top-row key 3 s; physical layout is numpad-like — dial top-left, 4×5 grid with one double-wide key (bottom-left) and one double-tall key (right column).

## 4. Code quality observations

- **Dead code** (verified no callers): `hid_parser._extract_handle_from_uuid`, `_should_check_combo_mapping` (always True), `EventType.DIAL_ROTATE/DIAL_CLICK` never emitted, `bluetooth_watcher._mac_to_dbus_path`, `device.py` HID UUID constants + `ALTERNATIVE_HID_SERVICES` (it subscribes to *everything* instead), `reconnect_attempts`/`max_reconnect_attempts` plumbing, `config.scan_timeout/reconnect_attempts/device_name/validate()/get_effective_config()`, four KeybindManager combo helpers (test-only), `device.get_device_info`.
- **Unused imports** in every module (AST-verified) — e.g. `struct`, `BleakScanner`, `socket`, `asdict`, `yaml`, `cast`.
- **Triplicated logic:** action-ID validation exists in 3 places with drift (bind, unbind, manager — only the manager checks duplicates); valid-button lists ×4; socket-path derivation ×2; list-bindings print loop ×3.
- **No-op logic:** `_build_capabilities` second loop can never add anything (looks up the same KEY_MAPPING it already fully added); capabilities are effectively static — and advertise all 225 keys + mouse buttons + REL axes regardless of bindings.
- **Misleading naming:** `_try_open_device` "waits for uinput device" — it *creates* one; the loop actually waits for `/dev/uinput` permission.
- **Doc drift:** CONTRIBUTING references `requirements.txt`, `debug_parser.py`, `tests/run_tests.py`, `make test-simple` (none exist) and "run with sudo" guidance contradicting the user-service design; tests/README documents obsolete formats; pyproject author is the `Your Name <your.email@example.com>` placeholder; NIXOS.md says "three command-line tools" and lists four.

## 5. Test suite assessment

- Tests **match the current implementation** — imports resolve, and hand-executed equivalents of every asserted scenario (combo peak-set, sticky press/release, dial CW/CCW/click, sensitivity 2.0, validation sets, config normalization) pass. The suite would very likely pass under pytest.
- **Stale fixtures:** `conftest.py` dial settings use `clockwise_key`/`counterclockwise_key`/`click_key` — names the implementation never reads (real: `DIAL_CW/CCW/CLICK`), so config-driven dial binding is untested; `INVALID_DATA` (0xFF×8) actually parses as buttons 13+14+15.
- **Hygiene:** KeybindManager fixtures `mkdir` into the real `~/.local/share` of whoever runs the tests.
- **Coverage gaps:** zero tests for `uinput_handler`, `device.py`, `bluetooth_watcher`, `config.save`/set-device/clear-device round-trip (would have caught H3/H4), socket framing (L1), CLI via CliRunner (would have caught M2), dial sensitivity < 1.0 (M5), type-1 byte-window limits (M6). Tests bind `KEY_CTRL` — an invalid key name (`KEY_LEFTCTRL` is correct) — reinforcing L3.

## 6. Security notes

- **Control socket:** `~/.local/share/huion-keydial-mini/control.sock`, no explicit `chmod` (umask-dependent), no auth, no size/connection limits, connections never closed server-side. Any process running as the user can rewrite bindings — acceptable for a personal tool, but the socket should move to `$XDG_RUNTIME_DIR` with dir 0700/sock 0600, and the GUI backend must add framing + timeouts.
- **Input-injection surface:** the virtual device always advertises every mapped key + mouse buttons + REL axes; combined with H2 (attach to arbitrary BLE peripherals) a hostile peripheral could feed the parser — the parser is defensive, so impact is limited to *bound* actions, but H2 should still be fixed.
- **Privilege model:** because the uaccess udev rule never matches (M4), installs fall back to blanket `input`-group membership — read access to **all** input devices (keylogging-adjacent). Fixing the rule lets users drop `input` group.
- **D-Bus:** handler filters interface/member/path but not `message.sender` (the AddMatch constrains it in practice; a defensive check is cheap).
- **systemd hardening** is mis-targeted (H1): effective hardening for a user unit = correct socket dir under `%t`, `Restart=on-failure`, drop the broken `Environment` line.

## 7. Environment prerequisites (this machine)

- No `pip`, no `ensurepip`/`venv`, no `uv`, no Node. Running the test suite or building a frontend requires (user-approved plan): `sudo apt install python3-venv python3-pip nodejs npm`.
- `python3-click` and `python3-yaml` exist system-wide; `evdev`, `bleak`, `dbus-next`, `pytest` do not.

## 8. Top opportunities ranked by GUI impact

1. **Persistence bridge** (H5+H3): `save`/`load` socket commands with a comment-preserving YAML writer (ruamel.yaml). Without it, a GUI "Apply" is a lie after reboot.
2. **Fix the out-of-box service** (H1+M1): repaired unit + removed phantom entry point. A GUI can't manage a service that doesn't start.
3. **Device identity & discovery** (H2+H4+M4): restore name/UUID filter, fix clear-device, MAC regex, add a scan-and-pick socket command → the GUI's pairing screen primitive.
4. **Dial sensitivity that works** (M5): detent accumulator (one event per 1/sensitivity detents).
5. **Truthful action model** (M3): implement scroll/relative actions or drop the claims; the GUI needs an accurate action-type enum.
6. **IPC hardening** (L1+M8+L3): newline framing, timeouts, single shared validator, key-name validation.
7. **Profiles**: the in-memory map + socket is 90% of a profile engine — add named binding-set save/switch server-side.
8. **Event stream**: `subscribe_events` socket command → "press a key to select it" UX + live monitor; replaces the broken `--test` mode (L5).
9. **Non-blocking, closeable uinput** (H6+M10): async retry, `.close()` on stop, static list-keys.
10. **Single-source version + working packages** (L7+M7): metadata-derived `__version__`, fixed deb/rpm, CI.
